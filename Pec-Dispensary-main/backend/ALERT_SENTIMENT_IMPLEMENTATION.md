# Alert Generation and Sentiment Analysis Implementation

## Overview

This implementation adds two major features to the dispensary system:
1. **Sentiment Analysis** - Using the trained BERT model to analyze patient feedback
2. **Alert Generation** - Anomaly detection for outbreak monitoring using Z-score, EWMA, and consecutive-day exceedance methods

## Database Schema Updates

### New Tables Created

1. **sentiment_analysis**
   - Stores sentiment analysis results from the ML model
   - Links to feedbacks table
   - Contains detected symptoms, sentiment scores, and aspect analysis

2. **outbreak_alerts**
   - Stores alerts generated by anomaly detection
   - Tracks severity (LOW, MEDIUM, HIGH, CRITICAL)
   - Tracks status (ACTIVE, ACKNOWLEDGED, RESOLVED)
   - Contains metadata about detection method and thresholds

3. **symptom_statistics**
   - Tracks daily symptom counts and statistics
   - Calculates baseline means, standard deviations, and EWMA values
   - Used for anomaly detection

## Implementation Details

### 1. Sentiment Analysis Service (`sentimentService.js`)

**Key Features:**
- Interfaces with the trained BERT model in `ml-model/infer.py`
- Uses Python subprocess to run sentiment analysis
- Automatically processes feedback when created
- Extracts:
  - Overall sentiment (positive/neutral/negative)
  - Detected symptoms with probabilities
  - Aspect sentiments (cleanliness, staff behavior, etc.)

**API Functions:**
- `analyzeSentiment(feedbackText)` - Runs ML model inference
- `processFeedbackSentiment(feedbackId, feedbackText)` - Full processing pipeline
- `getSentimentByFeedbackId(feedbackId)` - Retrieves saved sentiment data
- `getDoctorSentimentStats(doctorId, startDate, endDate)` - Aggregated statistics

### 2. Anomaly Detection Service (`anomalyDetectionService.js`)

**Anomaly Detection Methods:**

#### A. Z-Score Method (Z > 3)
- Calculates: `(current_value - baseline_mean) / baseline_std_dev`
- Triggers alert when Z-score > 3
- Severity levels:
  - Z > 5: CRITICAL
  - Z > 4: HIGH
  - Z > 3: MEDIUM

#### B. EWMA Deviation
- Exponentially Weighted Moving Average with alpha = 0.3
- Triggers when deviation > 2 × baseline_std_dev
- Severity levels:
  - Deviation > 3σ: HIGH
  - Deviation > 2σ: MEDIUM

#### C. Consecutive-Day Exceedance
- Monitors 3 consecutive days
- Each day must exceed: `baseline_mean + baseline_std_dev`
- Severity: HIGH

**API Functions:**
- `updateSymptomStatistics(symptom, date)` - Updates daily statistics
- `detectAnomaliesAndCreateAlerts(symptom, date)` - Runs all detection methods
- `processAllSymptomsForAnomalies(date)` - Processes all 10 symptoms
- `getActiveAlerts(filters)` - Retrieves active alerts
- `acknowledgeAlert(alertId, doctorId, notes)` - Marks alert as acknowledged
- `resolveAlert(alertId, doctorId, notes)` - Marks alert as resolved

### 3. ML Model Integration (`ml-model/run_inference.py`)

**Wrapper Script:**
- Reads JSON input from stdin
- Calls the trained BERT model's `predict_feedback()` function
- Returns JSON output with sentiment analysis results
- Handles errors gracefully

**Model Output Structure:**
```json
{
  "symptoms": ["symptom_fever", "symptom_headache"],
  "overall": "positive",
  "aspects": {
    "aspect_cleanliness": "negative",
    "aspect_staff_behaviour": "positive",
    ...
  },
  "symptom_probs": {
    "symptom_fever": 0.92,
    "symptom_headache": 0.85,
    ...
  }
}
```

## API Endpoints

### Feedback Endpoints (Enhanced)

**POST /api/feedbacks**
- Creates feedback and triggers sentiment analysis
- Automatically runs anomaly detection after processing

**GET /api/feedbacks/doctor/all**
- Returns all feedbacks for a doctor with sentiment analysis
- Query params: `startDate`, `endDate`, `sentiment`

**GET /api/feedbacks/doctor/sentiment-stats**
- Returns aggregated sentiment statistics
- Includes symptom breakdown and aspect breakdown
- Query params: `startDate`, `endDate`

**GET /api/feedbacks/:visitId**
- Returns feedback for a specific visit with sentiment data

### Alert Endpoints (New)

**GET /api/alerts/dashboard**
- Dashboard summary with alert counts by severity
- Shows top symptoms with active alerts
- Returns recent alerts count (last 7 days)

**POST /api/alerts/detect**
- Manually triggers anomaly detection for all symptoms
- Returns created alerts

**POST /api/alerts/detect/:symptom**
- Triggers detection for a specific symptom
- Example: `/api/alerts/detect/symptom_fever`

**GET /api/alerts**
- Lists all active alerts
- Query params: `symptom`, `severity`

**GET /api/alerts/history**
- Returns alert history
- Query params: `startDate`, `endDate`, `symptom`, `status`

**GET /api/alerts/:alertId**
- Returns specific alert details

**POST /api/alerts/:alertId/acknowledge**
- Marks alert as acknowledged
- Body: `{ "notes": "Investigating..." }`

- Marks alert as resolved
- Body: `{ "notes": "False alarm" }`

**GET /api/alerts/statistics/:symptom**

## Workflow

### Sentiment Analysis Workflow

1. Patient submits feedback via `POST /api/feedbacks`
2. Feedback is saved to database
3. If comments exist, `processFeedbackSentiment()` is called asynchronously
4. Python wrapper script is spawned with feedback text
5. ML model analyzes text and returns results
6. Results are saved to `sentiment_analysis` table
7. Feedback is marked as processed
8. Anomaly detection is triggered

### Anomaly Detection Workflow

1. `processAllSymptomsForAnomalies()` is called (triggered by feedback or manually)
2. For each symptom:
   - Updates daily statistics in `symptom_statistics` table
   - Calculates baseline mean and std dev from last 30 days
   - Calculates EWMA value
3. Runs three detection methods:
   - Z-score check
   - EWMA deviation check
   - Consecutive-day exceedance check
4. If anomaly detected, creates alert in `outbreak_alerts` table
5. Alert includes metadata about detection method and values
6. Prevents duplicate alerts within 24 hours

### Doctor Alert Monitoring Workflow

1. Doctor logs in and views dashboard
2. `GET /api/alerts/dashboard` shows overview
3. `GET /api/alerts` shows active alerts
4. Doctor can:
   - View alert details
   - Acknowledge alerts (marks as being investigated)
   - Resolve alerts (marks as handled)
   - View historical alerts
   - View symptom statistics

## Alert Severity Levels

- **CRITICAL** - Z-score > 5, immediate attention required
- **HIGH** - Z-score > 4 or consecutive-day exceedance detected
- **MEDIUM** - Z-score > 3 or EWMA deviation detected
- **LOW** - Minor deviations (reserved for future use)

## Alert Status Flow

1. **ACTIVE** - Alert created, requires attention
2. **ACKNOWLEDGED** - Doctor has seen and is investigating
3. **RESOLVED** - Issue handled or determined to be false alarm

## Performance Considerations

- Sentiment analysis runs asynchronously to avoid blocking feedback submission
- Anomaly detection uses efficient queries with proper indexing
- Statistics calculation limited to 30-day window
- Duplicate alert prevention to avoid spam

## Troubleshooting

### Sentiment Analysis Not Working
- Check Python installation: `python --version`
- Verify dependencies: `cd ml-model && pip list`
- Check model file exists: `ml-model/feedback_bert_symptom_head_v2.pt`
- Review logs for Python errors

### Alerts Not Generating
- Ensure sufficient historical data (at least 30 days)
- Check symptom statistics table is populated
- Verify symptom names match exactly
- Review threshold values in code

### Database Errors
- Ensure all migrations are run
- Check RLS policies are not blocking service role
- Verify foreign key relationships are intact
